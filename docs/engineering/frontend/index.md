# Frontend engineering — docs index

This file routes generic skills to the rule-bearing docs that apply for a given engineering task in this stack. Each section lists which docs to read; in some sections (Scaffold, Implementation pipeline) the section body also describes the procedure itself, because that procedure is frontend-specific.

Sibling rule docs in this directory:
- `stack.md` — pinned libraries per layer + rationale.
- `project-structure.md` — directory layout and organizational rules.
- `style.md` — TS strict flags, component idioms, naming.
- `state.md` — server vs client state, when Zustand is justified.
- `forms.md` — React Hook Form + Zod pairing.
- `api-integration.md` — `openapi-typescript` + `openapi-fetch` wiring, `src/api/` ownership.
- `routing.md` — React Router v7 data-router conventions.
- `testing.md` — Vitest + RTL + MSW conventions.
- `dependencies.md` — exact npm package list per concern.
- `deploy.md` — multi-stage Dockerfile + nginx config.

## Scaffold project

Read:
- `stack.md` — every pinned library and rationale.
- `project-structure.md` — directory layout the scaffold must produce.
- `dependencies.md` — exact npm package list per concern.
- `deploy.md` — Dockerfile and nginx config templates.

Conventions a fresh frontend project must satisfy:

- **Inputs the scaffold consumes:** project name (kebab-case), output directory, optional backend service name + backend OpenAPI path (used to configure the `gen:api` script for the project's `package.json`).
- **Top-level layout** (per `project-structure.md`): `src/{app,pages,features,components/ui,lib,api,styles,mocks}/`, `tests/e2e/`, `deploy/`, `public/`.
- **Required boilerplate under `src/`:**
  - `app/{App.tsx,providers.tsx,router.tsx}` — composition root (QueryClientProvider, RouterProvider, theme; data-router route table per `routing.md`).
  - `api/{client.ts,generated.ts}` — configured `openapi-fetch` instance and the codegen placeholder (regenerated by `gen:api`, never hand-edited).
  - `styles/globals.css` — Tailwind v4 entry + CSS variables.
  - `mocks/handlers.ts` — placeholder MSW handlers.
  - `main.tsx` — Vite entry that mounts `App` and imports `styles/globals.css`.
- **Required boilerplate under `tests/`:** `setup.ts` (loads `@testing-library/jest-dom`, starts the MSW server using handlers from `src/mocks/handlers.ts`).
- **Required boilerplate under `deploy/`:** `Dockerfile` and `nginx.conf` matching `deploy.md` verbatim.
- **Required boilerplate under `public/`:** `config.js` placeholder for runtime `window.__APP_CONFIG__` per `deploy.md`'s build-time-vs-runtime configuration pattern.
- **Required config files at the project root** (per `stack.md` and `dependencies.md`): `package.json` (with the pinned dependency set and the standard scripts — `dev`, `build`, `preview`, `test`, `test:watch`, `e2e`, `format`, `lint`, `typecheck`, `gen:api`, `prepare`), `tsconfig.json` (must enable `strict`, `noUncheckedIndexedAccess`, `verbatimModuleSyntax`, `noUnusedLocals`, `noUnusedParameters`, plus `@/* → src/*` path alias), `vite.config.ts` (React + Tailwind v4 plugins, `@ → src` alias, Vitest config with `happy-dom` and the `tests/setup.ts` setup file), `biome.json` (recommended rules, double quotes, semicolons, 2-space indent, 100-char width), `lefthook.yml` (pre-commit hooks for `biome check --staged` + `tsc --noEmit`). pnpm is the package manager — `pnpm-lock.yaml` is committed; no `package-lock.json` / `yarn.lock` may exist.
- **Post-scaffold hand-off conventions:** the runner must leave the project in a state where `tsc --noEmit` and `biome check .` both pass. Lefthook hooks must be installed (`lefthook install`) so the first commit runs the pre-commit pipeline. The user is then responsible for configuring `gen:api` if the backend OpenAPI path was not known at scaffold time, bringing in shadcn primitives via the shadcn CLI, adding the first real route under `src/pages/`, implementing MSW handlers for the first feature, and adding infra entries (per `deploy.md`) when the project joins an existing infra layout.

The runner that produces this layout — exact pnpm commands, file contents, etc. — is the responsibility of the scaffolding skill; this index does not prescribe its implementation.

## Write implementation

Read:
- `project-structure.md` — file placement, no barrels outside feature index, no `src/utils/`, `src/api/generated.ts` is codegen, tests co-located.
- `style.md` — TS strict, function components only, no `React.FC`, no implicit `any`.
- Plus the doc that matches the chunk's artifact:
  - `state.md` — chunk introduces client state.
  - `forms.md` — chunk introduces a form.
  - `api-integration.md` — chunk consumes the backend API.
  - `routing.md` — chunk adds or changes a route.

Bottom-up order (skip layers not in the chunk's `files`):

1. **Zod schemas + inferred types** (`src/features/<feature>/schemas.ts`) — schemas validate AND drive TS types (`z.infer<typeof X>`); single source of truth.
2. **API hooks** (`src/features/<feature>/api.ts`) — TanStack Query (`useQuery`, `useMutation`); call `apiClient` from `src/api/client.ts`; types from `src/api/generated.ts`; consistent query keys.
3. **Feature components** (`src/features/<feature>/components/`) — function components only; props inline or via local `type Props`; forms via RHF + `zodResolver`; Tailwind + shadcn primitives.
4. **Pages** (`src/pages/<route>/`) — `page.tsx` assembles components from `features/` and `components/`; ~80 lines max; loader in `loader.ts` if needed.
5. **Shared UI** (`src/components/`) — only when a second consumer demands it; do not pre-promote.
6. **App / routes / providers** (`src/app/`) — touch only when the chunk explicitly adds a new route or provider.

Build check: `pnpm tsc --noEmit` from `working_dir`. Do not run the test suite — that's the orchestrator's job.

## Write tests

Read:
- `testing.md` — Vitest + React Testing Library + MSW; AAA; query by user-perceivable attributes; `userEvent` not `fireEvent`.

For components, use `renderHook` for hooks. For Zod schemas, write pure data tests (parse valid / parse invalid). MSW handlers mock the network — no `fetch` stubs.

File placement: `<Name>.test.ts(x)` next to `<Name>.ts(x)`. E2E specs live under `tests/e2e/` and use Playwright; they are the only exception to co-location.

## Refactor project

Read every rule doc in this directory: `stack.md`, `project-structure.md`, `style.md`, `state.md`, `forms.md`, `api-integration.md`, `routing.md`, `testing.md`, `dependencies.md`, `deploy.md`.

Audit the entire project (every `.ts` / `.tsx` / `.css` file under `src/` and `tests/`, plus `package.json`, `tsconfig.json`, `vite.config.ts`, `biome.json`, `lefthook.yml`, `deploy/Dockerfile`, `deploy/nginx.conf`). For each violation record one row in the relevant category table.

Categories:
- **Structure** — `project-structure.md`.
- **State** — `state.md`.
- **Forms** — `forms.md`.
- **API** — `api-integration.md`.
- **Routing** — `routing.md`.
- **Testing** — `testing.md`.
- **Style** — `style.md`.
- **Tooling** — `stack.md` + `dependencies.md` (presence and shape of `biome.json`, `lefthook.yml`, `tsconfig.json` strict flags, `package.json` library pins, `pnpm-lock.yaml` committed and no `package-lock.json` / `yarn.lock`).

Per-violation row: `File:Line | Rule | Violation | Complexity (direct | behavioral)`.

Direct fixes (apply in-session): move files into `features/<feature>/`, delete stray barrel `index.ts`, relocate `utils/` content into `lib/`, move E2E specs into `tests/e2e/`, split overgrown pages by extracting feature components, replace raw `fetch`/`axios` with `apiClient`, replace hand-written payload types with imports from `src/api/generated.ts`, relocate query hooks to `features/<feature>/api.ts`, convert JSX `<Route>` to route-object form, remove `React.FC`, fix prop typing, add missing `biome.json` / `lefthook.yml`, patch `tsconfig.json` strict flags, remove `package-lock.json` / `yarn.lock`.

Behavioral fixes (carve out as a scoped follow-up feature and run through the standard feature pipeline — design, tests, implementation, verification):
- Form uses `useState` instead of RHF + Zod.
- Component duplicates server state into Zustand.
- Tests query by `getByTestId` instead of by role.
- Feature has no tests.

After all direct fixes: `pnpm biome check --write . && pnpm tsc --noEmit`.

## Implementation pipeline

Runs a chunk through 8 stages in order. Stage 0 is conditional; each remaining stage is either an engineering action (described below) or a shell command run from `working_dir`.

Pipeline-level constants:
- Test command: `pnpm vitest run --reporter=verbose`
- Type-check command: `pnpm tsc --noEmit`
- Format command: `pnpm biome check --write .`
- OpenAPI regen command: `pnpm openapi-typescript <path-to-upstream-api/openapi.yaml> -o src/api/generated.ts`

Stages:

| # | Name | Action | Retry cap |
|---|---|---|---|
| 0 | OpenAPI regen | Shell: OpenAPI regen command. Run only when the chunk depends on a chunk whose `files` include `api/openapi.yaml`. | 0 (precondition — on failure stop) |
| 1 | Tests | Write tests for the production code. Constraint: production files do not exist yet — tests must compile against the types/exports the spec promises; runtime failure is expected at this stage. | 0 |
| 2 | Verify tests | Audit the tests for convention compliance per `testing.md`. Loop: on violations re-write tests with findings prepended. | 1 |
| 3 | Implementation | Write production code that makes the existing tests pass. Constraint: test files are FROZEN — never modify any `*.test.ts(x)` or `*.spec.ts(x)` file. | 0 (retries via stages 4/5) |
| 4 | Run tests + type-check | Shell: test command followed by type-check command. On failure: diagnose and fix production code (test-freeze constraint verbatim), re-run both. | 2 |
| 5 | Verify logic | Audit that the implementation matches the chunk's intent. On fail: re-run Implementation with findings prepended (test-freeze verbatim), re-run stages 4 and 5. | 1 |
| 6 | Format | Shell: format command. On non-zero exit (Biome could not auto-fix all issues): stop and report. | 0 |
| 7 | Verify style | Audit the implementation against `style.md` + `project-structure.md`. On violations: apply each fix (no test edits, no scope expansion; run the format command after fixing), then re-audit. | 1 |

Test-freeze rule (applies to stages 3, 4 retry, 5 retry, 7 fix): never modify any `*.test.ts(x)` or `*.spec.ts(x)` file. If the only correct fix would require changing a test, stop and report.

Scope rule (all stages): only files listed in the chunk's `files` may be touched. If a stage needs a file outside that list, that is a spec gap — escalate.

Final report shape (pipeline label `Frontend`): 8-row table with Stage / Status / Notes, plus the list of files created/modified and any blocking issues. Stage 0 row shows status `skipped` when the precondition does not hold.
