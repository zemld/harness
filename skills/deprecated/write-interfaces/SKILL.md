# Write Interfaces

Given a description of a feature or service flow, produce the Go files that define the contracts: domain entities, the incoming service interface, and outgoing port interfaces. Do not write concrete implementations, struct bodies, or stub methods.

## Reference

File placement follows `docs/engineering/go/service-structure.md`.

## What to produce

### Domain entities — `internal/domain/<context>/entities/<entity>.go`

Pure Go structs with no imports from `internal/`. One top-level type per file, named after the entity. Enums, ID types, and value objects each get their own file. Types shared across bounded contexts go in `internal/domain/shared/`.

### Incoming service interface — `internal/domain/<context>/service.go`

```go
package <context>

import "context"

type Service interface {
		Creator
		Canceller
}

type Creator interface {
    CreateOrder(ctx context.Context, req CreateOrderRequest) (Order, error)
}

type Canceller interface {
    CancelOrder(ctx context.Context, id OrderID) error
}
```

- Methods are business actions, not HTTP verbs
- First argument is always `context.Context`
- Returns `(T, error)` or `error` only
- Request/response types that are not domain entities live in the same file or package

### Repository port — `internal/ports/repository/<name>.go`

```go
package repository

import (
    "context"
    "<module>/internal/domain/<context>/entities"
)

// Repository is the aggregate — used only in app.go for DI.
type Repository interface {
    Saver
    ByIDGetter
}

type Saver interface {
    Save(ctx context.Context, order entities.Order) error
}

type ByIDGetter interface {
    ByID(ctx context.Context, id entities.OrderID) (entities.Order, error)
}
```

### Client port — `internal/ports/clients/<name>.go`

```go
package clients

import "context"

type PaymentClient interface {
    Charge(ctx context.Context, req ChargeRequest) (ChargeResult, error)
}
```

## How to proceed

1. If the feature description is vague, ask one focused question to pin down: bounded context name, entities involved, operations the service exposes, external systems touched (database? external API?).
2. Propose the file tree you plan to create. If the bounded context or operations are ambiguous, pause and clarify before writing.
3. Write files in order: entities → incoming service interface → repository port → client port.

## Rules

- `ports/` contains interfaces only — no structs, no implementations
- Skip any port the feature doesn't need (no repository port if there's no persistence; no client port if there are no external calls)
- One interface per repository operation + aggregate `Repository` embedding all (used only in `app.go`)
- No stutter: package `order`, not `OrderService`; package `perfume`, not `PerfumeRepository`
- Do not redeclare interfaces from another package — import and use them
- Do not write handler implementations — REST handlers are generated by ogen from `api/openapi.yaml`; if REST endpoints are needed, note which operations to add to the spec
