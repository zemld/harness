# Go Service Structure

```
<service>/
├── api/                        # API specs (openapi.yaml)
├── cmd/main.go                 # Entry point — calls into internal/app, no logic; holds //go:generate for ogen
├── config/                     # Static config files (config.yaml)
├── deploy/                     # Dockerfile and deployment assets
├── generated/                  # Generated code (protobuf, grpc) — never edit by hand
├── migrations/                 # SQL migrations, timestamped
├── internal/
│   ├── adapters/               # Concrete I/O implementations only (no interfaces here)
│   │   ├── brokers/
│   │   │   └── kafka/          # franz-go: producer, consumer client, serde (one broker per service)
│   │   ├── clients/
│   │   │   └── <name>/         # One directory per outbound client
│   │   │       └── <impl>/     # Concrete implementation (e.g. aistudio/, http/)
│   │   ├── repository/
│   │   │   └── <name>/         # One directory per persistence scope
│   │   │       └── <impl>/     # Concrete implementation (e.g. postgres/, ydb/)
│   │   └── transactor/
│   │       └── postgres/       # Transactor impl + tx-in-context helper (begin/commit/rollback, querier)
│   ├── agents/                 # Agent definitions (LLM orchestration), by bounded context
│   │   └── <bounded_context>/  # Same context as domain/ and services/
│   │       ├── agent.go        # Coordinator: model + instruction + sub-agents; wiring only
│   │       ├── instruction.go  # System prompt (or //go:embed instruction.md)
│   │       └── <sub_context>/  # One sub-agent per sub-context it covers
│   │           ├── agent.go
│   │           ├── instruction.go
│   │           ├── tools/      # LLM tool-call → domain service call (one tool per file)
│   │           └── agent_test.go
│   ├── api/                    # Inbound transport handlers
│   │   ├── grpc/
│   │   │   ├── convert/
│   │   │   ├── handler/
│   │   │   └── server/
│   │   ├── kafka/              # Inbound consumer transport (mirrors rest/grpc)
│   │   │   ├── convert/        # kafka message ↔ domain (+ Schema Registry serde)
│   │   │   ├── handler/        # One handler per topic / event type
│   │   │   └── server/         # Consumer-group runner, Start/Stop
│   │   └── rest/
│   │       ├── ogen/           # Generated by ogen — never edit by hand
│   │       ├── convert/        # Domain ↔ ogen-type conversions
│   │       ├── handler/        # Implements ogen.Handler interface (one method per operation)
│   │       └── server/         # Thin wrapper: calls ogen.NewServer(handler)
│   ├── app/
│   │   ├── app.go              # Composition root — fx.New(...); only place with concrete types
│   │   └── <layer>.go          # Optional: fx.Module per layer, add only when app.go exceeds ~40 lines
│   ├── config/
│   │   ├── load.go
│   │   └── types.go
│   ├── domain/                 # Domain models + incoming service interfaces
│   │   ├── <bounded_context>/
│   │   │   ├── service.go      # Incoming service interface (contract for callers)
│   │   │   └── entities/       # Business entities, value objects, invariants
│   │   └── shared/             # Models used across multiple bounded contexts
│   ├── ports/                  # Outgoing port interfaces (what services need from I/O)
│   │   ├── clients/
│   │   │   └── <name>.go
│   │   ├── producers/
│   │   │   └── <name>.go       # EventPublisher — what a service publishes to Kafka
│   │   ├── repository/
│   │   │   ├── <name>.go
│   │   │   └── outbox.go       # Outbox.Add — persist events in the same tx as domain state
│   │   └── tx/
│   │       └── transactor.go   # Transactor — run several repository writes in one transaction
│   ├── services/               # Business logic implementations
│   │   └── <name>/
│   │       ├── service.go          # Struct + constructor only
│   │       ├── <operation>.go      # One public method per file
│   │       └── <operation>_test.go
│   ├── workers/                # Background loops (ticker-driven); thin Start/Stop runner over a service
│   │   └── <name>/
│   └── utils/                  # Generic, domain-free helpers (last resort)
├── go.mod
├── Makefile
├── .golangci.yml
├── .mockery.yml
```

## REST API generation

`api/openapi.yaml` is the single source of truth for the REST contract. `ogen` consumes it and generates everything under `internal/api/rest/ogen/`: request/response types, the `Handler` interface, routing, request parsing, response encoding, and validation.

**`cmd/main.go`** holds the generation directive (paths are relative to `cmd/`):

```go
//go:generate go run github.com/ogen-go/ogen/cmd/ogen --target ../internal/api/rest/ogen --package ogen --clean ../api/openapi.yaml
```

Placing it in `cmd/main.go` keeps the root free of orphaned `package main` files that break `go build ./...`.

**Makefile `gen` target** must run `go generate ./...` (in addition to any protobuf generation).

**Developer responsibilities after generation:**

- Implement each method of the generated `Handler` interface in `internal/api/rest/handler/` — one file per operation, following the same one-operation-per-file rule as services.
- Write domain ↔ ogen-type conversions in `internal/api/rest/convert/`.
- `internal/api/rest/server/` calls `ogen.NewServer(handler)` and wraps it in a struct with `Start`/`Stop` methods.

**What disappears with ogen:** there is no `dto/` package. All request and response types come from `ogen/`. Never write DTO structs by hand for REST endpoints.

## Health & operational endpoints

Liveness/readiness probes (`GET /health`) are **operational, not business logic**, so they stay **out of `api/openapi.yaml` and out of ogen**. Never model `/health` as an OpenAPI operation.

A health check is a hand-written handler:

- `internal/api/rest/handler/health.go` — a `HealthHandler` with `ServeHTTP` (writes `200 OK`, empty body) and `Route() string` returning `"GET /health"`.
- `internal/api/rest/server/` mounts it on a plain `http.ServeMux`. For fx services it's provided as an fx component and started via `fx.Lifecycle`; for manually-wired services it exposes `Start()`/`Stop(ctx)`.

**A service whose only REST surface is `/health` has no ogen layer at all** — no `api/openapi.yaml`, no `api/ogen.yaml`, no `internal/api/rest/ogen/`, no ogen dependency or `//go:generate` directive. Its business API is gRPC (and/or Kafka); the HTTP server exists only to answer probes. ogen is introduced only when the service gains real REST business operations.

The gateway is the reference for the handwritten health handler (it also composes ogen business routes with handwritten SSE/proxy passthroughs on the same mux).

## Layering

**`domain/` has zero imports from `internal/`.** No framework types, no adapter types.

**`services/` depends on `domain/` types and `ports/` interfaces only.** Never import a concrete adapter package from a service. Never import another `internal/services/` package from a service — if service A needs service B, it must depend on B's domain interface, not on `internal/services/B` directly.

**Only `internal/app/app.go` may import concrete types** and wire them together via DI.

**Port interfaces (outgoing contracts) live in `ports/`.** `ports/` contains interfaces only — no structs, no implementations. Examples: HTTP clients, repositories. Do NOT define service-to-service contracts here.

**Incoming service interfaces live in `domain/<bounded_context>/service.go`.** Every service that other code depends on must declare its interface in `domain/` — not in `internal/services/<name>/interfaces.go`. This applies to all callers: REST handlers, agents, other services. Example: `domain.INNSearchService`, `domain.RelevanceService`. The implementation lives in `internal/services/`; the contract lives in `domain/`.

## Agents

Agents are LLM-driven orchestrators — a layer of their own beside `services/`, obeying the same inward-pointing dependency rule. An agent definition is wiring only (model, instruction, tools, sub-agents); it holds no business logic.

**Organize agents by bounded context.** `internal/agents/<context>/` uses the same contexts as `domain/` and `services/`: a coordinator covers the context, each sub-agent covers a sub-context in a nested package. Sub-agents never import each other — a coordinator delegates, and sub-agents communicate through shared session state.

**Tools are inbound, like a handler.** A tool translates an incoming tool-call into a `domain.<Service>` call and converts the result back. Tools import `domain/` interfaces only — never a concrete adapter or an `internal/services/<name>` package. One tool per file under the agent's `tools/`.

**The service drives the agent.** The generic executor that runs an agent is built in `app.go` and injected. The loop that turns an agent's output stream into a domain result is business logic, so it lives in the service that implements `domain.<Service>` and holds the injected executor — this is how an agent hides behind a domain interface. Do not wrap the executor in a port; there is nothing to swap.

**A model that does no I/O is a service, not an adapter.** A model implementation that only shapes a request, calls a client port, and parses the response is a service (`internal/services/<model>/`); it works with any provider client returning the same response contract. The provider client that makes the actual call is an outbound client — `adapters/clients/<provider>/` behind a `ports/clients/` port.

**Framework interfaces are exempt from the `domain/` contract rule.** Agents, tools, and custom model implementations satisfy interfaces the framework owns, not ours — so their contracts are not declared in `domain/` (that rule is for our own service contracts).

## Kafka & messaging

Kafka plays **two independent roles**, and each lands on a different side of the architecture. The domain never imports the Kafka client — exactly as it never imports ogen or a DB driver.

| Role | Direction | Analogous to | Location |
|------|-----------|--------------|----------|
| **Consumer** | inbound (driving) | REST/gRPC handler | `internal/api/kafka/` |
| **Producer** | outbound (driven) | HTTP client | `ports/producers/` + `adapters/brokers/kafka/` |

There is one broker per service, so `adapters/brokers/kafka/` has no `<name>` level — unlike `clients/` and `repository/`.

### Consumer (inbound)

Mirrors `rest/`: `server/` runs the consumer-group loop and holds `Start`/`Stop`, `handler/` has **one handler per topic / event type** (same one-operation-per-file rule as services), `convert/` maps `kafka message ↔ domain` and owns Schema Registry (de)serialization. A handler that fails to process after its retry budget publishes the message to a **dead-letter topic** via a producer — it never blocks the partition.

### Producer (outbound)

A service depends on an `EventPublisher` interface in `ports/producers/<name>.go`, never on `franz-go`. The concrete producer lives in `adapters/brokers/kafka/`.

## Delivery guarantees: outbox

Direct produce/consume is the default. Use the **outbox** pattern only when an event must not be lost if the process dies between the DB commit and the publish (event is a source of truth for another service).

**The service owns the transaction, not the repository.** Writing domain state and enqueuing the event must be atomic, so the caller of both ports wraps them in one transaction. A repository must never import another adapter, so it cannot own this — the service does, through a `Transactor`.

**`Transactor` (`ports/tx/transactor.go`)** exposes `Do(ctx, func(ctx) error) error`. It begins a transaction, stores it in `ctx`, runs the callback, then commits (or rolls back on error). The service calls its ports with that `ctx`; each repository pulls the ambient transaction from `ctx` (falling back to the pool when absent). The concrete type lives in `adapters/transactor/postgres/`, which also owns the tx-in-context key and `querier` helper — the only place a DB transaction type appears. Services stay free of DB types; `repository/` and `outbox` never depend on each other.

**Layout of the three pieces:**

- Table → `migrations/` (an `outbox` table with a `published_at` column).
- Write port → `ports/repository/outbox.go` (`Outbox.Add`), impl in `adapters/repository/outbox/postgres/`.
- Relay → in-process, not a sidecar. Logic (fetch unpublished batch → publish via the producer port → mark published) is an ordinary service in `internal/services/outbox/`; a thin ticker runner with `Start`/`Stop` lives in `internal/workers/outbox/` and is started from `app.go`. `internal/workers/` is the general slot for any ticker-driven background loop.
