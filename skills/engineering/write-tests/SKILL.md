Spawn a single subagent and hand it the intent plus the instructions below. Do not execute any of these steps inline — delegate everything to the subagent.

## Subagent instructions

You receive an intent describing what logic needs tests. Work through these steps in order, waiting for each to complete before moving to the next.

**1. Read the testing conventions**

Read `docs/engineering/go/testing.md`. Every test you write must follow these rules exactly — structure, mocking approach, assertion style, parallelism.

**2. Locate the code under test**

Search the codebase to find the files that implement the logic described in the intent. Read them to understand:
- Public method signatures and return types
- Domain types and their fields
- External dependencies (repositories, clients, third-party APIs)

**3. Enumerate test cases**

Invoke the `case-analyzer` skill, passing the intent and the code you read as context. Wait for the full case table before writing any code. The table covers every scenario — happy path, edge cases, error cases, corner cases.

**4. Write the tests**

Write one `TestXxx` function per public method. Each case from the table becomes one row in the test table. Follow the conventions from step 1:

- Table-driven structure even for a single case
- `t.Parallel()` at the top of the test function and at the top of each `t.Run` subtest
- AAA layout (Arrange / Act / Assert) with a blank line between each section
- `require` for assertions that must stop the test on failure; `assert` for non-fatal checks
- Mocks generated by mockery — never hand-written; if the needed mock doesn't exist yet, state which interface requires a `mockery` re-run before the tests can compile

**5. Place the files**

Put each `_test.go` file next to the file it tests. Use the `_test` package suffix (external test package).
